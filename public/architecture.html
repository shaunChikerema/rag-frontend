04 — Design Decisions

Why pgvector over Pinecone?
Single source of truth — vectors and metadata live in the same Postgres database as everything else. No extra service, lower latency on retrieval, and the match_documents RPC keeps the cosine similarity math in the DB engine, not Python. Generous free tier on Supabase covers this project's scale.
Why separate task types for embed vs retrieve?
Gemini's embedding model was trained with RETRIEVAL_DOCUMENT for chunks stored at ingest time, and RETRIEVAL_QUERY for the user's question at query time. Using the correct task type measurably improves cosine similarity scores — the vectors are optimised for asymmetric search, not just nearest-neighbour.
Why temperature 0.2, not 0?
Temperature 0 produces technically correct but often robotic phrasing. 0.2 allows natural language variation while keeping the model citation-bound to the context window — the system prompt enforces inline [1][2] citations mapped to the sources array, so hallucination is structurally prevented, not just hoped for.
Why 200-char chunk overlap?
Sentences that straddle two chunk boundaries appear in full in at least one chunk. Without overlap, a key fact split across chunks would be truncated in both — retrieval would return broken context. 200 chars (~50 tokens) is enough to preserve a complete sentence without duplicating so much content that storage balloons.
Why Groq for generation?
Groq's LPU hardware runs Llama 3.3 70B at ~800 tokens/sec — fast enough that the demo feels responsive without streaming. Free tier is production-usable for a portfolio project. The model itself is open-weight and strong at instruction-following and citation formatting.
